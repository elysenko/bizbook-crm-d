### `backend/` (template-backend)
Standalone NestJS + Prisma + PostgreSQL REST API with JWT auth, user CRUD, Swagger docs,
and a `docker-compose.yml` that spins up a local Postgres instance. Listens on port 3000.

### `web/` (template-web)
A separate full-stack bundle, not a single service:
- `web/frontend/` — Angular 17 SPA (project name `frontend`), login + home pages, JWT
  stored in `localStorage`, HTTP interceptor attaches `Authorization: Bearer`, route
  guard on `/home`. Deep-linkable routes declared via `data.flow` in `app.routes.ts`.
  `AppComponent` carries `data-testid="app-ready"` (added by the scaffolder to match
  the platform's render-gate convention — this attribute did not exist in the raw
  template and must be preserved).
- `web/backend/` — a second, separate NestJS + Prisma API (Postgres), built to run
  alongside the Angular SPA behind the bundled `web/nginx.conf`, which proxies `/api/`
  to a `backend:3000` upstream.
- `web/Dockerfile.frontend`, `web/Dockerfile.backend`, `web/nginx.conf` — two-image
  build (nginx static host for the SPA + node runtime for its API).

**Note on duplication:** the requested `backend` stack entry and `web`'s own nested
`web/backend/` are independent NestJS+Prisma services from different templates — this
is a property of the platform's template set, not a scaffolder decision. Downstream
deploy/build agents should treat `backend/` and `web/backend/` as two distinct
services unless the plan says otherwise.

## Next steps for the developer / coder agent
1. **backend/**: `cd backend && pnpm install`, copy `.env.template` values (none shipped
   with this template — create `.env` with `DATABASE_URL`, `JWT_SECRET`, `JWT_EXP`,
   see `src/common/config/env.config.ts` for the full list), `docker compose up -d` to
   start Postgres locally, then `npx prisma migrate dev`.
2. **web/backend/**: `cd web/backend && npm install`; configure its own `DATABASE_URL`
   env var (see `prisma/schema.prisma`), run Prisma migrations before first boot.
3. **web/frontend/**: `cd web/frontend && npm install && npm start` (proxies `/api` via
   `proxy.conf.json` in dev). Build features on top of the existing login/home/guard
   scaffold per the plan (BizBook: clients, services, appointments, dashboard, revenue).
4. Do not remove `data-testid="app-ready"` from `web/frontend/src/app/app.component.ts`
   — the post-deploy render gate waits on it. See `.colossus-acceptance.json`.
5. `colossus.yaml` at the project root documents the build/serve contract for deploy
   agents — keep `outputDir`/ports in sync if directory layout changes.
